# Debug

이 이슈를 근본 원인까지 추적하여 디버깅하라.

## 핵심 원칙

### 1. 소스 코드를 직접 읽어라
관련된 부모/래퍼 컴포넌트의 소스 코드를 직접 읽어라 — 동작 방식을 추측하지 마라.
함수가 어떤 인자를 받는지, 컴포넌트가 props를 어떻게 처리하는지, 미들웨어가 요청을 어떻게 변환하는지 등을 실제 코드에서 확인해야 한다. 문서나 이름만 보고 추측하면 잘못된 방향으로 빠지기 쉽다.

### 2. 소스 수준에서 수정하라
소비자(consumer) 수준이 아닌 소스(source) 수준에서 수정하라.
문제의 원인이 되는 코드 자체를 고쳐야 한다. 호출하는 쪽에서 우회하거나 방어 코드를 추가하는 것은 임시방편일 뿐, 근본적인 해결이 아니다.

### 3. 인과 관계 체인을 먼저 보여줘라
구현에 들어가기 전에 근본 원인에서 증상까지의 인과 관계 체인을 보여줘라.
수정 코드를 작성하기 전에, 문제가 어디서 시작되어 어떤 경로를 거쳐 사용자가 관찰한 증상으로 이어지는지 전체 흐름을 명확히 정리해야 한다.

## 디버깅 워크플로우

### 1단계: 증상 파악
- 에러 메시지, 스택 트레이스, 재현 조건을 수집한다
- 문제가 발생하는 정확한 지점을 특정한다

### 2단계: 역추적
- 증상이 나타나는 코드에서 시작하여 호출 체인을 거슬러 올라간다
- 각 단계에서 관련 소스 코드를 직접 읽는다
- 부모 컴포넌트, 래퍼, 미들웨어, 유틸리티 함수 등을 빠짐없이 확인한다

### 3단계: 인과 관계 체인 정리
수정 전에 반드시 아래 형식으로 인과 관계를 정리한다:

```
[근본 원인] A에서 X가 잘못 처리됨
    ↓
[전파 경로] B가 A의 결과를 그대로 사용
    ↓
[전파 경로] C가 B의 잘못된 값을 기반으로 동작
    ↓
[증상] 사용자가 관찰한 문제
```

### 4단계: 소스 수준 수정
- 근본 원인이 있는 코드를 직접 수정한다
- 소비자 측 우회가 아닌 소스 측 수정을 한다
- 수정 후 인과 관계 체인의 각 단계가 올바르게 동작하는지 검증한다
