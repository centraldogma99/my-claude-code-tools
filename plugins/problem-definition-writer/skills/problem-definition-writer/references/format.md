# 문제 정의서 포맷

## 메타데이터 헤더 (문서 최상단)

| 항목 | 내용 |
|------|------|
| 작성일 | YYYY-MM-DD |
| 상태 | Draft / Approved |
| 작성자 | 이름 |
| 관련 기획 문서 | 링크 (선택) |

---

## 필수 블록 (순서대로)

### 1. `## 배경`

왜 이 작업이 필요한가. 현재 상황과 문제의 맥락을 자유 서술한다. 비어있으면 안 된다.

### 2. `## 해결 기준`

무엇이 충족되면 문제가 해결된 것인가. 목록 형식으로 작성한다.

- 각 항목은 **참/거짓으로 판단 가능한 문장**이어야 한다.
- 최소 1개 이상 필수.
- "UX를 개선한다", "더 나은 경험을 제공한다" 같은 방향성 서술 금지.

### 3. `## 규칙과 제약 (선택)`

해당하는 경우에만 작성한다. 섹션 자체 생략 허용.

- 포함 시: 특정 조건에서 금지되는 동작, 불변 조건 등 선언형 서술 허용.
- "해당 없음" 리터럴도 허용.
- 규칙이 핵심인 작업에서는 이 블록이 본문이 된다.

### 4. `## 범위 밖`

이번 작업에서 명시적으로 하지 않는 것을 선언한다. 비어있으면 안 된다.

- 선언형/명사형 서술 허용 (예: "컬럼 A, B는 제거한다").
- 뒤에서 혼란을 예방하는 가장 강력한 장치다.

### 5. `## 변경 이력`

문서 수정 시 이력을 남긴다. 각 엔트리에 날짜 / 변경 내용 / 사유 3요소 필수.

- 초기 작성 시 빈 테이블 또는 "(최초 작성)" 엔트리 허용.
- 테이블 또는 목록 형식 모두 허용.

---

## 블록별 quality 규칙

| 블록 | quality 규칙 |
|---|---|
| 배경 | 비어있지 않음, 서술 허용 |
| 해결 기준 | 각 항목이 T/F 판단 가능해야 함, 최소 1개 |
| 규칙과 제약 (선택) | 섹션 생략 또는 "해당 없음" 허용, 포함 시 선언형 서술 허용 |
| 범위 밖 | 선언형/명사형 서술 허용, 비어있지 않음 |
| 변경 이력 | 엔트리 있다면 각각 날짜+변경내용+사유 3요소 필수, 비어있거나 "(최초 작성)" 허용 |

---

## 금지 사항

- ID 체계 (G1, BR1 등) 도입 금지
- 우선순위 태그 (P0/P1 등) 도입 금지
- 구체적인 기능 명세, 화면 흐름, UI 세부사항, 구현 방법, 기술 스택은 이 문서의 범위 밖 (How/Detail 층위)

---

## 예시 스니펫 — 백오피스 컬럼 케이스

소스 문서의 백오피스 개선 사례(`문제 정의서를 도입합시다.md:86-98`)를 이 포맷으로 표현하면:

**`## 범위 밖` 작성 방식:**

```markdown
## 범위 밖
- 이번 작업의 범위에 컬럼 변경은 포함되지 않는다.
```

또는 컬럼 제거가 명시적 의도인 경우:

```markdown
## 범위 밖
- 컬럼 A, B는 사용성 개선을 위해 제거한다.
```

이 한 줄이 있었다면, 개발자는 피그마에서 컬럼 누락을 발견했을 때 즉시 판단할 수 있었다. 문제 정의서의 핵심 가치는 이처럼 **판단 근거를 문서화**하는 데 있다.
