# 문제 정의서에 넣으면 안 되는 것

문제 정의서는 **Why/What** 층위의 문서다. 여기에 How(기능 명세, 플로우)나 Detail(UI, 코드, 기술 스택)이 섞이면, 기획 문서를 수정할 때 문제 정의서가 판단 기준 역할을 하지 못한다. 아래 카테고리는 모든 블록에서 금지하거나, 블록별로 조건부 적용한다.

## 개발/구현 상세

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 코드, 타입, 인터페이스 | `interface Props { ... }`, `onSelect(agent, areaLabel)` | 문제 정의서는 "무엇"이지 "어떻게"가 아님 |
| 파일 경로, 컴포넌트명 | `AgentSelectModal.tsx`, `DraftEditPanel` | 구현 결정은 개발 문서에 |
| API 엔드포인트 상세 | `GET /v2/polis/instead/...` | API 스펙 문서에 분리 |
| DB 스키마, 쿼리 | `SELECT * FROM allocations` | 기술 설계서에 |
| 라이브러리/프레임워크명 | `React Query`, `Zustand`, `antd` | 기술 스택은 개발 문서에 |
| 아키텍처 패턴 | `View/Container 패턴`, `Presentational Pattern` | 기술 설계서에 |

## 이전 버전과의 비교

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 기존 플랜/설계 참조 | "기존 플랜 대비", "v1에서는..." | 문제 정의서는 독립적으로 읽을 수 있어야 함 |
| 변경 전/후 비교 (구현 수준) | "suggestedDate 배너 제거", "conflict 쿼리 2곳 중복 → 서버 필터링" | 구현 마이그레이션 내용 |
| 제거 대상 목록 | "DraftEditPanel의 독립 DatePicker 제거" | 리팩토링 계획 |

## 내부 프로세스

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 설계 결정 과정 | "Architect가 ITERATE 판정", "3명 리뷰어 검토 결과" | 의사결정 로그는 ADR에 |
| 대안 비교 분석 | "Option A vs Option B vs Option C" | 설계 문서에 |
| 회의록/토론 내용 | "~라고 논의했음" | 별도 기록 |

## 기능/솔루션 명세 (문제 정의서 고유)

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 구체적 기능 명세 | "필터 드롭다운을 추가한다", "로그인 실패 시 토스트를 띄운다" | How의 영역 — 기획 문서에 |
| UI/화면 흐름 | "목록 화면에서 상세로 이동한다", "모달이 뜬다" | Detail의 영역 — 디자인 문서에 |
| 우선순위 태그 | `P0`, `P1`, `P2` | 기능 명세 메타데이터 — 기획 문서에 |
| 측정 수치 | "전환율 15% 향상", "DAU 1만 달성" | 성공 지표는 기획 문서에 |

## 판별 기준 — 5-question discriminator

문장을 읽고 아래 질문 중 하나라도 "예"이면 문제 정의서에 넣지 않는다.

1. **개발자만 이해할 수 있는가?** → 구현 상세
2. **이전 버전을 모르면 의미가 없는가?** → 비교/마이그레이션
3. **"어떻게 만들지"에 대한 답인가?** → 기획/기술 문서
4. **특정 기술 스택에 종속되는가?** → 기술 문서
5. **특정 기능/화면을 전제하는가?** → 기획/디자인 문서 (문제 정의서 고유)

> **동기화 메모**: Q1-Q4는 `prd-writer`의 discriminator와 동일하다. 이 파일 최상단 `SYNC-WITH` 주석에 따라 두 플러그인의 Q1-Q4는 동기 유지되어야 한다. Q5는 문제 정의서 고유 질문으로, Why/What 층위에서 특정 솔루션을 전제하는 서술을 걸러낸다.

## Exclusion × Block 적용 매트릭스

어떤 질문을 어떤 블록에서 적용할지 명시한다. 이것은 block-aware validation의 핵심 규칙이다.

| 질문 | 배경 | 해결 기준 | 규칙과 제약 | 범위 밖 |
|---|:---:|:---:|:---:|:---:|
| Q1 (개발자 전용) | apply | apply | apply | apply |
| Q2 (이전 버전 의존) | apply | apply | apply | apply |
| Q3 (How의 답) | apply | apply | **skip** | **skip** |
| Q4 (기술 스택) | apply | apply | apply | apply |
| Q5 (기능/화면 전제) | apply | apply | **skip** | **skip** |

### 매트릭스 설명

- **"규칙과 제약" 블록**은 특정 조건에서의 금지/허용 규칙을 담으므로, 선언형 서술(예: "외부 파트너 계정은 본 기능을 호출할 수 없다")이 자연스럽다. 이 블록에서는 Q3/Q5를 skip한다. 단 Q1/Q2/Q4는 여전히 적용 — 규칙도 기술 스택 용어로 쓰면 안 된다.
- **"범위 밖" 블록**은 "이번 작업에서 하지 않을 것"을 명시하는 선언형 블록이다. 소스 예시의 "컬럼 A, B는 사용성 개선을 위해 제거한다" 같은 서술은 How로 보일 수 있으나 범위 밖 선언으로 정당하다. Q3/Q5 skip.
- **다른 블록(배경, 해결 기준)**에서는 전체 5-question을 엄격히 적용한다.

### 항상 금지 (모든 블록)

아래 카테고리는 블록과 무관하게 모든 블록에서 금지한다. 매트릭스의 skip 규칙이 적용되지 않는다.

- 개발/구현 상세 (코드, 파일 경로, API, DB, 라이브러리, 아키텍처 패턴)
- 이전 버전과의 비교
- 내부 프로세스
- UI/화면 흐름
- 우선순위 태그
- 측정 수치
