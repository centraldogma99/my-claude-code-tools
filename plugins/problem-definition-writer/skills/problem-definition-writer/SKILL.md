# Problem Definition Writer

문제 정의서 작성과 검토를 오케스트레이션한다. 문제 정의서는 **Why/What** 층위의 문서로, PRD(기획 문서, How) **위에** 얹힌다. 기획·디자인·구현이 수정될 때 "맞다/틀리다"를 판단할 상위 기준 역할을 한다.

**핵심 원칙: 작성과 검증은 반드시 분리한다.** 같은 컨텍스트에서 작성하고 검증하면 확증 편향이 발생하므로, 검증은 항상 별도 서브에이전트(`problem-definition-writer:problem-definition-validator`)가 문서와 `references/` 규칙 파일만 보고 수행한다.

## 작성 주체 원칙

문제 정의서의 작성 주체는 일반적으로 **PM**이다. 무슨 문제를 어떻게 해결해야 하는지 가장 잘 이해하는 사람이 쓰는 것이 원칙이고, 대부분의 경우 그 사람은 PM이다.

이 스킬은 **PM을 대체하지 않는다.** PM의 입력(아이디어, 메모, 미팅 노트, 초안)을 받아 구조화하는 도구다.

- PM과 협업하여 **PM의 언어로** 기록한다.
- 대신 작성하지 말고 PM 입력을 구조화한다.
- PM 입력이 불충분하면 추측으로 채우지 말고 PM에게 질문한다.

## 워크플로우

### 작성 모드 (새 문제 정의서)

입력(PM 아이디어, 요구사항 메모, 미팅 노트 등)을 표준 문제 정의서로 변환한다.

1. **입력 분석** — 배경 / 해결 기준 / 규칙·제약(있는 경우) / 범위 밖 후보를 식별한다.
2. **포맷 적용** — [references/format.md](references/format.md)의 블록 구조를 그대로 따른다:
   - `## 배경`
   - `## 해결 기준`
   - `## 규칙과 제약 (선택)` — 해당 없으면 섹션 자체 생략
   - `## 범위 밖`
   - `## 변경 이력` — 초기 작성 시 "(최초 작성)" 엔트리 허용
3. **How 침투 자가 점검** — 각 문장을 쓸 때마다 **"구현 방식이 바뀌어도 이 문장은 유효한가?"** 자문한다. 아니면 한 단계 추상화한다:
   - "토스트를 띄운다" → "사용자가 실패 원인을 인지할 수 있다"
   - "필터 드롭다운을 추가한다" → "운영자가 조건별로 데이터를 찾을 수 있다"
4. **해결 기준 품질 점검** — "개선한다 / 향상한다 / 강화한다" 같은 방향성 서술은 구체화한다. 참/거짓으로 판단 가능한 문장으로 변환한다.
5. **제외 항목 자가 점검** — [references/exclusions.md](references/exclusions.md)의 **5-question discriminator**와 **× block 매트릭스**를 적용해 금지 카테고리(코드, 파일 경로, 기능 명세, UI 흐름, 우선순위 태그 등)가 본문에 들어가지 않도록 한다.
6. **파일 저장** — 사용자가 지정한 경로 또는 합리적 기본 경로(예: `docs/problem-definition/<slug>.md`)에 저장한다. 저장 위치를 절대 경로로 기록한다.
7. **검증 에이전트에 위임** — 아래 "검증 에이전트 호출" 섹션의 호출 규약을 그대로 따른다.
8. **검증 결과 보고** — 에이전트가 반환한 구조화 결과를 사용자에게 그대로 전달한다. 전체 판정이 FAIL이면 "수정 권고" 섹션을 기준으로 다음 액션을 논의한다.

### 검토 모드 (기존 문제 정의서)

사용자가 기존 문제 정의서 경로를 주면 작성 과정을 건너뛰고 바로 검증한다.

1. **파일 경로 확인** — 절대 경로로 정규화한다.
2. **검증 에이전트에 위임** — 호출 규약 따르기.
3. **결과 보고** — 에이전트 출력을 그대로 전달한다.

### 엔트리 추가 모드 (변경 이력 업데이트)

사용자가 기존 문제 정의서에 변경을 반영해달라고 요청할 때:

1. 기존 파일을 읽는다.
2. 해당 본문 블록을 수정한다.
3. `## 변경 이력`에 새 엔트리를 추가한다. 날짜 / 무엇을 바꿨는지 / 왜 바꿨는지 **3요소 필수**.
4. 저장 후 검증 에이전트에 위임한다.

## 검증 에이전트 호출

검증은 **반드시 별도 서브에이전트**로 실행한다. 이 에이전트는 대화 컨텍스트를 공유하지 않으며, 문서와 `references/` 규칙 파일만으로 판단한다. 직접 이 스킬 안에서 검증 체크리스트를 수행하지 말고, 반드시 아래처럼 Agent 툴로 위임한다.

```
Agent(
  prompt="<아래 템플릿 참조>",
  subagent_type="problem-definition-writer:problem-definition-validator"
)
```

### 프롬프트 템플릿

위임할 때 전달하는 프롬프트는 다음 두 가지 정보만 포함하면 충분하다. 에이전트 본문에 이미 전체 검증 절차·체크리스트·출력 포맷이 내장되어 있으므로 중복 지시하지 않는다.

```
Validate the problem definition document at the following path. Judge solely from the document and the reference rules — do not assume any conversation context.

Problem definition file: <문서의 절대 경로>
Reference rules directory: <이 플러그인의 references 디렉토리 절대 경로, 예: /.../plugins/problem-definition-writer/skills/problem-definition-writer/references>
```

레퍼런스 디렉토리는 이 스킬과 같은 폴더에 위치한 `references/`를 절대 경로로 넘긴다.

### 결과 전달 규칙

- 에이전트의 출력은 그대로 사용자에게 전달한다. 요약하거나 재구성하지 않는다.
- FAIL 항목이 있으면 "수정 권고" bullet을 기준으로 사용자에게 어떤 항목을 먼저 수정할지 묻는다.
- 작성 모드에서 FAIL이 나오면 사용자 지시에 따라 문서를 수정한 뒤, **다시 한 번** 같은 에이전트에 위임해 재검증한다. 이 루프는 사용자가 중단을 지시하거나 PASS가 나올 때까지 반복한다.
- 작성 중간에 스킬이 스스로 "이 정도면 괜찮아 보인다"는 식으로 자체 승인을 하지 않는다. 판정 권한은 전적으로 검증 에이전트에 있다.

## 링크 전파 가이드 (문서 운영의 핵심)

작성 또는 수정이 완료되면 반드시 사용자에게 안내한다:

> **생성된 문제 정의서의 경로를 관련 PRD, Figma 파일, Jira 티켓에서 링크로 참조하세요.** 어떤 산출물을 보든 한 번의 클릭으로 그 위에 있는 Why/What에 접근할 수 있어야, 문제 정의서가 판단 기준으로 실제 기능한다.

혼자 조용히 문제 정의서를 고치고 공유하지 않는 상태가 가장 위험하다. 수정 시에도 관련 기획자·디자이너·개발자에게 변경을 알리도록 사용자에게 상기시킨다.

## 언제 이 스킬을 사용하지 않는가

문제 정의서는 **일정 규모 이상의 신규 프로젝트**에만 적용한다. 아래 경우에는 오버헤드만 만든다:

- 단순 버그 수정
- 작은 UI 개선 (문구 조정, 아이콘 교체 등)
- 문구 변경
- 여러 직군이 얽히지 않는 개인 작업

"일정 규모 이상"의 판단은 PM의 재량이다. 여러 직군이 얽히고 의사결정이 반복될 가능성이 있는 작업에만 적용한다.
