# PRD에 넣으면 안 되는 것

## 개발/구현 상세

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 코드, 타입, 인터페이스 | `interface Props { ... }`, `onSelect(agent, areaLabel)` | PRD는 "무엇"이지 "어떻게"가 아님 |
| 파일 경로, 컴포넌트명 | `AgentSelectModal.tsx`, `DraftEditPanel` | 구현 결정은 개발 문서에 |
| API 엔드포인트 상세 | `GET /v2/polis/instead/...` | API 스펙 문서에 분리 |
| DB 스키마, 쿼리 | `SELECT * FROM allocations` | 기술 설계서에 |
| 라이브러리/프레임워크명 | `React Query`, `Zustand`, `antd` | 기술 스택은 개발 문서에 |
| 아키텍처 패턴 | `View/Container 패턴`, `Presentational Pattern` | 기술 설계서에 |

## 이전 버전과의 비교

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 기존 플랜/설계 참조 | "기존 플랜 대비", "v1에서는..." | PRD는 독립적으로 읽을 수 있어야 함 |
| 변경 전/후 비교 (구현 수준) | "suggestedDate 배너 제거", "conflict 쿼리 2곳 중복 → 서버 필터링" | 구현 마이그레이션 내용 |
| 제거 대상 목록 | "DraftEditPanel의 독립 DatePicker 제거" | 리팩토링 계획 |

## 내부 프로세스

| 금지 항목 | 예시 | 이유 |
|----------|------|------|
| 설계 결정 과정 | "Architect가 ITERATE 판정", "3명 리뷰어 검토 결과" | 의사결정 로그는 ADR에 |
| 대안 비교 분석 | "Option A vs Option B vs Option C" | 설계 문서에 |
| 회의록/토론 내용 | "~라고 논의했음" | 별도 기록 |

## 판별 기준

문장을 읽고 아래 질문에 "예"이면 PRD에 넣지 않는다:

1. **개발자만 이해할 수 있는가?** → 구현 상세
2. **이전 버전을 모르면 의미가 없는가?** → 비교/마이그레이션
3. **"어떻게 만들지"에 대한 답인가?** → 기술 설계서
4. **특정 기술 스택에 종속되는가?** → 기술 문서
