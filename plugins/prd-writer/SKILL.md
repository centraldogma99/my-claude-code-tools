# PRD Writer

PRD 작성, 검토, 정합성 검증을 수행한다.

**핵심 원칙: 작성과 검증은 반드시 분리한다.** 같은 컨텍스트에서 작성하고 검증하면 확증 편향이 발생하므로, 검증은 항상 별도 에이전트가 PRD 문서만 보고 수행한다.

## 워크플로우

### 작성 모드 (새 PRD)

입력(아이디어, 요구사항 목록, 기획 메모 등)을 표준 PRD로 변환한다.

1. 입력 분석 — 목표, 사용자, 기능 식별
2. 포맷 적용 — [references/format.md](references/format.md) 참조
3. 제외 항목 검증 — [references/exclusions.md](references/exclusions.md) 참조
4. PRD 파일 작성 및 저장
5. **검증 에이전트 실행** — 아래 "검증 에이전트" 섹션 참조
6. 검증 결과를 사용자에게 보고

### 검토 모드 (기존 PRD)

기존 PRD 파일 경로를 받아 검증 에이전트에 위임한다.

1. PRD 파일 경로 확인
2. **검증 에이전트 실행** — 아래 "검증 에이전트" 섹션 참조
3. 검증 결과를 사용자에게 보고

## 검증 에이전트

검증은 **반드시 별도 에이전트**로 실행한다. 이 에이전트는 대화 컨텍스트를 공유하지 않으며, PRD 문서와 검증 기준만으로 판단한다.

```
Agent(
  prompt=아래 프롬프트 템플릿 사용
)
```

### 검증 에이전트 프롬프트 템플릿

```
PRD 정합성 검증을 수행하라. 대화 맥락 없이, PRD 문서 자체만으로 판단하라.

## 검증 대상
{PRD 파일 경로}

## 검증 기준
다음 두 파일을 읽고 기준으로 사용하라:
- 포맷 기준: {skills 경로}/references/format.md
- 제외 항목 기준: {skills 경로}/references/exclusions.md

## 검증 체크리스트

### 1. 포맷 준수
format.md의 필수 섹션이 모두 존재하는가?

### 2. 제외 항목 위반
exclusions.md에 정의된 금지 항목(코드, 파일 경로, API 엔드포인트, 라이브러리명 등)이 포함되어 있는가?

### 3. 목표 ↔ 기능 요구사항
- 모든 목표에 대응하는 기능 요구사항이 있는가?
- 어떤 목표에도 연결되지 않은 기능 요구사항이 있는가?

### 4. 기능 요구사항 ↔ 사용자 시나리오
- 모든 P0 기능 요구사항이 최소 1개 시나리오에서 커버되는가?
- 시나리오에서 언급되지만 기능 요구사항에 없는 동작이 있는가?

### 5. 비즈니스 규칙 ↔ 기능 요구사항
- 비즈니스 규칙이 기능 요구사항과 모순되는가?
- 기능 요구사항이 비즈니스 규칙을 전제하지만 명시되지 않은 것이 있는가?

### 6. Acceptance Criteria ↔ 전체
- 모든 목표에 대응하는 AC가 있는가?
- AC가 검증 가능한가? (예/아니오로 판단 가능)
- AC와 기능 요구사항/비즈니스 규칙 사이에 모순이 있는가?

### 7. 시나리오 간 일관성
- 시나리오 A에서 가능한 것이 시나리오 B에서 불가능하지 않은가?
- 동일 개념에 다른 용어를 사용하지 않는가?

## 출력 형식

## PRD 검토 결과

### 포맷
- [PASS/FAIL] 항목별 결과

### 제외 항목 위반
- [위반 없음] 또는 위반 목록

### 정합성
- [PASS/FAIL] 체크 항목별 결과 + 모순이 있으면 상세 설명
```
